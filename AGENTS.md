# AGENTS.md

## Project Overview

This repository contains the public website for Financial Decisions, an independent financial planning firm. It is an Astro 5 site with a small content collection for newsletters and a mostly handcrafted CSS layer shaped by the design brief in `DESIGN.md`.

The site is trust-first, editorial, and conservative in tone. Changes should make the experience feel more established and more premium, not more playful, startup-like, or feature-heavy.

## Stack

- Astro 5
- Tailwind CSS v4, imported through `src/styles/global.css`
- Mostly static `.astro` pages and layouts
- Astro Content Collections for newsletter entries in `src/content/blog`
- PDF.js assets served from `public/` for optional inline PDF viewing

## Commands

Run all commands from the repo root.

- `npm run dev` starts the local Astro dev server on port `4321`
- `npm run build` builds the production site into `dist/`
- `npm run preview` previews the built site

There is no formal automated test suite in the repo right now. The minimum verification step for most changes is `npm run build`.

## Source Of Truth

Use these files before making design or architecture decisions:

- `DESIGN.md`: canonical design direction for the homepage and overall visual language
- `src/styles/global.css`: global tokens, shared UI patterns, button styles, reveal behavior, and page shell rules
- `src/layouts/Layout.astro`: global wrapper, header/footer composition, Google Analytics tag, and reveal-on-scroll script
- `src/content.config.ts`: schema for newsletter content entries

Note: `README.md` is still the default Astro starter README and is not an accurate project guide.

## Repository Map

- `src/pages`: route files for marketing pages and archive pages
- `src/components`: shared Astro components such as header, footer, hero, and PDF viewer
- `src/layouts`: shared page shells and newsletter post layout
- `src/styles`: global and page-specific stylesheet files
- `src/content/blog`: newsletter markdown entries
- `public/images`: brand photography and newsletter images
- `public/insights-assets/newsletters`: downloadable newsletter PDFs
- `public/pdfjs-dist` and `public/pdf.worker.min.mjs`: PDF.js browser assets

## Architecture Notes

- The primary site shell uses `src/layouts/Layout.astro`.
- Most pages rely on global styles plus page-local `<style>` blocks inside `.astro` files.
- Tailwind is available, but the site is not built as a utility-first codebase. Prefer existing CSS patterns over introducing large new utility-only sections.
- The newsletter archive at `src/pages/insights/index.astro` reads from the `blog` collection and promotes one featured issue.
- Individual newsletter pages are generated by `src/pages/insights/newsletters/[slug].astro` and rendered through `src/layouts/BlogPostLayout.astro`.

## Design Rules

Follow `DESIGN.md`. The short version:

- Preserve a calm, established, editorial feel
- Use the warm neutral palette already defined in CSS custom properties
- Keep the single bronze accent; do not introduce extra accent colors
- Avoid fintech styling, generic SaaS cards, loud gradients, glow effects, or trendy startup language
- Use serif selectively for emphasis, not everywhere
- Keep motion subtle and purposeful; prefer opacity and transform only
- Maintain asymmetric desktop layouts and strict single-column mobile collapse
- Prefer real firm photography over decorative illustrations

When touching UI, work with the existing tokens:

- `--accent`
- `--accent-dark`
- `--accent-soft`
- `--accent-muted`
- `--page-max-width`
- `--page-content-max-width`

## Content Conventions

### Standard Pages

- Keep copy precise, calm, and professional
- Avoid marketing filler and banned words called out in `DESIGN.md`
- Maintain accessibility basics: meaningful alt text, real headings, real buttons/links, and visible focus states

### Newsletters

Newsletter entries live in `src/content/blog/*.md` and should follow the schema in `src/content.config.ts`.

Required practical fields:

- `title`
- `pubDate`
- `readingTime`

Commonly used optional fields:

- `description`
- `image`
- `featured`
- `draft`
- `pdfPath`
- `issueItems`

When adding or updating a newsletter:

- Keep exactly one item featured unless there is a deliberate reason to change that behavior
- Store the downloadable PDF in `public/insights-assets/newsletters`
- Store supporting images under `public/images/newsletters/...`
- Use site-relative paths in frontmatter such as `/images/...` and `/insights-assets/...`
- Preserve the existing editorial presentation used in current markdown files

Some newsletter content uses inline HTML and table-based layouts inside markdown. If you refactor those files, verify mobile behavior carefully because `BlogPostLayout.astro` contains special mobile rules to stack older table layouts.

## Styling Guidance

- Put shared tokens and reusable patterns in `src/styles/global.css`
- Keep page-specific styling inside the relevant `.astro` file unless it is clearly reusable
- Reuse existing classes like `page-shell`, `page-container`, `page-title`, `ui-card`, `btn`, and `reveal`
- Do not add animation that ignores `prefers-reduced-motion`
- Do not remove the existing restrained hover and reveal system unless replacing it with something equally conservative and accessible

## JavaScript Guidance

- Default to static Astro markup
- Add client-side JavaScript only when the interaction truly needs it
- If you add JS, keep it small and progressively enhanced
- Preserve the existing inline reveal script behavior in `Layout.astro` unless there is a clear bug or performance reason to change it

## Analytics And Third-Party Assets

- Google Analytics is loaded in `src/layouts/Layout.astro`
- PDF.js worker and module files are served from `public/`
- Be cautious about adding new third-party scripts; this site should stay lightweight and trustworthy

## Verification Checklist

Before finishing meaningful changes:

- Run `npm run build`
- Check affected routes for layout regressions
- Check mobile behavior when editing newsletter content or page-level grids
- Verify links to PDFs and images still resolve from `public/`
- Confirm motion still respects reduced-motion expectations

## Agent Preferences

- Do not rely on `README.md` for project context
- Treat `DESIGN.md` as the visual brief unless the user explicitly overrides it
- Prefer small, surgical edits over broad rewrites
- Preserve existing content tone and firm credibility
- When unsure, optimize for clarity, restraint, and trust
